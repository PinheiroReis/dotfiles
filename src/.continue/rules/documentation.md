# Documentation Guidelines

## Read and Follow Project Documentation

Always read and strictly follow the guidelines defined in the project's documentation files before making any suggestions, changes, or contributions.

## Priority Documentation Files

The following files contain critical project information and must be consulted:

### 1. **CONTRIBUTING.md**
- Contribution guidelines and workflow
- Code submission process
- Branch naming conventions
- PR requirements and templates

### 2. **CODE_OF_CONDUCT.md**
- Community standards and expectations
- Behavioral guidelines
- Enforcement policies

### 3. **README.md**
- Project overview and purpose
- Quick start guide
- Basic usage examples
- Project structure

### 4. **INSTALLATION.md** / **SETUP.md**
- Installation steps and requirements
- Environment setup
- Dependencies and prerequisites
- Configuration instructions

### 5. **TESTING.md**
- Testing strategy and requirements
- How to run tests
- Writing new tests
- Coverage expectations

### 6. **SECURITY.md**
- Security policies
- Vulnerability reporting process
- Security best practices
- Supported versions

### 7. **ARCHITECTURE.md** / **DESIGN.md**
- System architecture and design decisions
- Component interactions
- Technical decisions and rationale

### 8. **CHANGELOG.md** / **HISTORY.md**
- Version history
- Breaking changes
- Migration guides

### 9. **LICENSE**
- Licensing terms
- Usage restrictions
- Attribution requirements

### 10. **API.md** / **docs/** directory
- API documentation
- Detailed technical specifications
- Integration guides

## Additional Documentation

Also check for:
- `.github/` directory for templates and workflows
- `docs/` directory for extended documentation
- Wiki pages if available
- Any project-specific `.md` files

## Workflow

1. **Before starting any work**: Read relevant documentation files
2. **When making suggestions**: Ensure they align with documented guidelines
3. **When writing code**: Follow documented standards and conventions
4. **When uncertain**: Reference documentation or ask for clarification
5. **When documentation is missing**: Point out the gap and suggest creating it

## Important Notes

- Documentation takes precedence over assumptions
- If documentation conflicts with requests, highlight the conflict
- Always respect documented processes and standards
- Suggest documentation updates when guidelines are outdated or unclear
